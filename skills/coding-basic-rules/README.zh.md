# Coding Basic Rules — Claude Code 12 条工程纪律规则

一个 `SKILL.md` 文件，为 Claude Code 的行为强制执行 12 条基础工程纪律规则。

## 问题所在

LLM 编程助手有一套共通的失效模式：

> 它们假设而不是询问。明明 50 行能搞定，它们非要过度设计。它们会动不该动的代码。工作没做完就宣告完成。它们混合相互矛盾的模式而不是暴露冲突。它们悄悄超出 Token 预算和上下文限制。

这些不是模型缺陷——而是行为默认值。规则可以覆盖它们。

---

## 解决方案

12 条原则集中在一个文件中，直接针对这些失效模式：

| 规则 | 解决的问题 |
|------|-----------|
| **编码前思考** | 默默假设、隐藏困惑、缺少权衡 |
| **简洁优先** | 过度设计、投机性功能、过早抽象 |
| **精准修改** | 无关编辑、未经请求的重构、风格漂移 |
| **目标驱动执行** | 漫无目的地按步骤执行、进度无法验证 |
| **模型只用于判断** | 在确定性代码能解决的地方使用 LLM |
| **Token 预算是硬限制** | 悄悄超出上下文、丧失连贯性 |
| **暴露冲突而非平均** | 默默混合模式、代码库不一致 |
| **写之前先读** | 不理解调用方/导出就添加代码 |
| **测试验证意图** | 业务逻辑被破坏时测试仍能通过 |
| **每步都要检查点** | 状态未跟踪、任务中途失去线索 |
| **遵循代码库约定** | 以个人品味为由悄悄分叉 |
| **大声失败** | 默默跳过、虚假的"已完成"、测试被略过 |

## 12 条规则详解

### 规则 1 — 编码前思考

**不要假设。不要隐藏困惑。呈现权衡。**

- 明确说明假设——如果不确定，询问而不是猜测
- 当存在歧义时，呈现多种解释
- 当存在更简单的方案时，提出异议
- 困惑时停下来——在继续之前说清楚哪里不明白

### 规则 2 — 简洁优先

**用最少的代码解决问题。不做推测性工作。**

- 不添加要求之外的功能
- 不为一次性代码创建抽象
- 不添加未被请求的"灵活性"
- **检验标准：** 资深工程师会说这过于复杂吗？如果是，简化。

### 规则 3 — 精准修改

**只动必须动的。只清理自己造成的混乱。**

- 不"改进"相邻的代码、注释或格式
- 不重构没有问题的东西
- 匹配现有风格，即使你有不同偏好
- **检验标准：** 每一行修改都必须能直接追溯到用户的请求。

### 规则 4 — 目标驱动执行

**定义成功标准。循环验证直到达成。**

不要按步骤盲目执行，而是将任务转化为可验证的目标：

| 不要这样... | 转化为... |
|------------|----------|
| "修复这个 bug" | "编写重现 bug 的测试，然后让它通过" |
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "重构 X" | "确保重构前后测试都能通过" |

强有力的成功标准让模型能够自主循环执行。弱标准需要不断澄清。

### 规则 5 — 模型只用于判断调用

- 使用模型做：分类、起草、摘要、信息提取
- 不使用模型做：路由、重试、确定性变换
- 如果代码能回答，就用代码回答

### 规则 6 — Token 预算不是建议

- 单任务上限：4,000 tokens。单会话上限：30,000 tokens。
- 如果接近预算，先汇总，再开新对话
- **主动暴露超限。不要悄悄超出。**

### 规则 7 — 暴露冲突，而非取平均

- 如果两种模式相互矛盾，选一个（更近的 / 更经过验证的）
- 说明理由，标记另一个待清理
- 不要将冲突模式混合成一个混血方案

### 规则 8 — 写之前先读

- 在添加代码之前，阅读导出项、直接调用方、共享工具
- "看起来是正交的"很危险
- 如果不确定代码为何如此结构，先询问

### 规则 9 — 测试验证意图，而非仅验证行为

- 测试必须编码行为**为何**重要，而不只是**是什么**
- 一个业务逻辑被破坏时仍能通过的测试是错误的

### 规则 10 — 每个重要步骤后都要检查点

- 总结已完成什么、已验证什么、还剩什么
- 不要从一个你无法描述清楚的状态继续
- 如果失去线索，停下来重新陈述

### 规则 11 — 遵循代码库约定，即使你不认同

- 在代码库内部，合规性 > 个人品味
- 如果某个约定确实有害，说出来——不要悄悄另起炉灶

### 规则 12 — 大声失败

- 如果有任何内容被默默跳过，"已完成"就是错误的
- 如果有任何测试被跳过，"测试通过"就是错误的
- 默认暴露不确定性，而不是隐藏它

## 安装

**选项 A：Claude Code 插件市场（推荐）**

在 Claude Code 中执行两条命令即可完成安装：

```
/plugin marketplace add Adamatoma/cowork-skills
/plugin install coding-basic-rules@cowork-skills
```

安装完成后，在 Claude Code 中输入 `/coding-basic-rules` 即可调用。

**选项 B：手动安装**

克隆仓库并将 skill 复制到 Claude Code skills 目录：

```bash
git clone https://github.com/Adamatoma/cowork-skills.git
cp -r cowork-skills/skills/coding-basic-rules/skills/coding-basic-rules ~/.claude/skills/coding-basic-rules
```

安装完成后，在 Claude Code 中输入 `/coding-basic-rules` 即可调用。

## 如何判断规则在起作用

如果你看到以下情况，说明规则正在发挥作用：

- **diff 中多余的改动更少** —— 只有被请求的修改出现
- **因过度复杂而导致的重写更少** —— 代码第一次就写得简洁
- **澄清问题在实现之前提出** —— 而不是在犯错之后
- **干净、精简的 PR** —— 没有顺带的重构或"顺手改进"

---

## 权衡说明

这些规则偏向**谨慎而非速度**。对于琐碎任务（简单拼写错误、显而易见的单行修改），请自行判断——并非每个改动都需要完整的严谨流程。

目标是减少非琐碎工作中代价高昂的错误，而不是拖慢简单任务。
